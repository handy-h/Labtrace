## 2. 严重问题（P0）

### 2.1 `backup.go:49-54` — `os.Create` 错误被丢弃，nil 指针 panic

```go
tmpFile, _ := os.Create(tmpPath)   // 错误被丢弃
defer tmpFile.Close()              // 若 Create 失败，tmpFile == nil → panic
defer os.Remove(tmpPath)
_, err = tmpFile.ReadFrom(file)    // panic
```

**修复**：

```go
tmpFile, err := os.Create(tmpPath)
if err != nil {
    c.JSON(http.StatusInternalServerError, models.Error("创建临时文件失败"))
    return
}
```

### 2.2 `imaging.go:625` — goroutine 内 `config.Load()` 错误被丢弃，nil cfg → panic

`imaging.go:625`（`ReOCRImagingReport` 的异步 goroutine 内）：

```go
cfg, _ := config.Load()                       // 错误丢弃，cfg 可能为 nil
ocrResults, err := services.Recognize(fileBytes, cfg)  // panic
```

goroutine 逃逸出 Gin 的 `Recovery` 中间件范围，panic 会直接导致进程崩溃。

同类问题（错误被丢弃，但在请求 goroutine 内，相对安全）：

| 文件 | 行号 | Handler |
|------|------|---------|
| `ocr.go` | 54 | `Upload` |
| `imaging.go` | 70 | `UploadImagingReport` |
| `backup.go` | 22 | `ExportBackup` |
| `backup.go` | 59 | `ImportBackup` |
| `batch_import.go` | 179 | `ConfirmBatchImport` |

**根本修复**：将 `cfg` 在 `main.go` 启动时作为依赖注入，而不是每次请求重新 Load。最小修复方案——每处改为：

```go
cfg, err := config.Load()
if err != nil {
    c.JSON(http.StatusInternalServerError, models.Error("配置加载失败"))
    return
}
```

## 3. 高优先级问题（P1）

### 3.1 `main.go:166-173` — `os.Exit(0)` 强杀 OCR goroutine，绕过 defer

```go
go func() {
    <-sig
    fmt.Println("\nShutting down...")
    database.Close()
    os.Exit(0)  // defer database.Close()（main 第 30 行）不会执行
                // 正在运行的 OCR goroutine 被强杀，可能写入已关闭的 DB
}()
```

`os.Exit` 不执行任何 `defer`。正在异步执行 OCR 的 goroutine（`ocr.go:86`、`imaging.go:103`、`imaging.go:617`）会在 DB 关闭后继续写入 `database.DB`，导致 "use of closed database" 错误或数据损坏。

**修复**：改用 `http.Server.Shutdown(ctx)` 优雅关闭，并用 `sync.WaitGroup` 跟踪 OCR goroutine 生命周期：

```go
srv := &http.Server{Addr: ":" + cfg.Port, Handler: r}
go func() {
    sig := make(chan os.Signal, 1)
    signal.Notify(sig, syscall.SIGINT, syscall.SIGTERM)
    <-sig
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    srv.Shutdown(ctx)
}()
srv.ListenAndServe()
// defer database.Close() 在此正常执行
```

## 4. 中优先级问题（P2）

### 4.1 `ocr_quota.go:51-58`（`ReOCR` handler）— OCR 失败返回 HTTP 200

```go
if err != nil {
    database.DB.Exec(`UPDATE lab_reports SET ocr_status = 'failed' WHERE id = ?`, id)
    c.JSON(http.StatusOK, models.Success(gin.H{  // 应为 5xx
        "status": "failed",
        "error":  err.Error(),
    }))
    return
}
```

客户端无法通过 HTTP 状态码判断失败。应改为 `http.StatusBadGateway`（502，OCR 外部服务失败）。

### 4.2 更新/删除不存在资源返回 HTTP 200，应返回 404

下列 handler 调用 `Exec` 后不检查 `RowsAffected`，对不存在的 ID 静默返回 200：

| 文件 | 行号 | Handler |
|------|------|---------|
| `subject.go` | 104–113 | `UpdateSubject` |
| `subject.go` | 115–123 | `DeleteSubject` |
| `subject.go` | 169–176 | `UpdateHospital` |
| `subject.go` | 178–186 | `DeleteHospital` |
| `testitem.go` | 74–82 | `UpdateTestItem` |
| `testitem.go` | 85–93 | `DeleteTestItem` |

**修复模板**（以 `UpdateSubject` 为例）：

```go
result, err := database.DB.Exec(
    `UPDATE subjects SET name=?, gender=?, birth_date=?, updated_at=datetime('now') WHERE id=?`,
    s.Name, s.Gender, s.BirthDate, id,
)
if err != nil {
    c.JSON(http.StatusInternalServerError, models.Error(err.Error()))
    return
}
if n, _ := result.RowsAffected(); n == 0 {
    c.JSON(http.StatusNotFound, models.Error("subject not found"))
    return
}
c.JSON(http.StatusOK, models.Success(nil))
```

## 6. 中间件顺序 ✅

`main.go:43-44`：

```go
r := gin.Default()       // 内置 Logger + Recovery
r.Use(middleware.CORS()) // CORS 追加在 Recovery 之后
```

执行顺序：`Logger → Recovery → CORS → Handler`。CORS 中间件在调用 `c.Next()` 之前已写入响应头，即使 Handler panic 后 Recovery 返回 500，CORS 头依然有效。顺序正确，无需修改。
