## 一、N+1 查询问题

### 🔴 问题1：`GetTrendData` 循环内单独查参考区间

**位置**：`internal/services/trend_service.go:101-111`

主查询返回 N 条 `report_items`，对每条有 `ref_interval_id` 的记录都单独发起一次查询：

```go
rows, _ := database.DB.Query(query, args...)
for rows.Next() {
    // ...
    if refID.Valid {
        // N次！每条记录单独查
        database.DB.QueryRow(`SELECT value_min, value_max FROM reference_intervals WHERE id = ?`, refID.Int64).Scan(...)
    }
}
```

**修复方案**：将 `reference_intervals` JOIN 进主查询：

```sql
SELECT ri.id, COALESCE(ri.test_item_id, 0), ...,
    ref.value_min, ref.value_max
FROM report_items ri
JOIN lab_reports lr ON lr.id = ri.report_id
LEFT JOIN hospitals h ON h.id = lr.hospital_id
JOIN subjects s ON s.id = lr.subject_id
LEFT JOIN test_items ti ON ti.id = ri.test_item_id
LEFT JOIN reference_intervals ref ON ref.id = ri.ref_interval_id  -- 新增 JOIN
WHERE lr.subject_id = ? AND lr.ocr_status = 'imported'
ORDER BY lr.sample_date ASC, ri.test_item_id ASC
```

然后在 `rows.Scan` 时直接扫描 `ref.value_min, ref.value_max`，删除循环内的 `QueryRow`。

**位置**：`internal/handlers/report.go:338-352`

```go
rows := database.DB.Query(`SELECT ... FROM report_items WHERE report_id = ?`, reportID)
```

### 🔴 问题3：`MatchTestItemByName` 全表扫描 + 循环中重复调用

**位置**：`internal/services/testitem_service.go:65-83`

步骤4 直接 `SELECT id, standard_name FROM test_items` 拉取全表，然后在 Go 层做字符串匹配。且每次调用最多执行 4~5 次串行查询（步骤1~5）。在 `matchRefAndCalcFlag` 和 `ConfirmBatchImport` 的循环中被重复调用，形成严重的 N×M 次查询。

```go
rows, err := database.DB.Query(`SELECT id, standard_name FROM test_items`) // 全表！
// 然后用 Go 代码遍历做 normalizeName 比较
```

**修复方案**：在调用方将整张 `test_items` 表一次性加载进内存 map，传入匹配函数：

```go
// 在 matchRefAndCalcFlag 开头一次性加载
type testItemRow struct { id int64; name string }
var allItems []testItemRow
rows, _ := database.DB.Query(`SELECT id, standard_name FROM test_items`)
for rows.Next() {
    var r testItemRow
    rows.Scan(&r.id, &r.name)
    allItems = append(allItems, r)
}
rows.Close()

// 循环内传入 allItems 而非重复查库
for i := range items {
    matchID := matchByNameInMemory(items[i].TestItemName, allItems)
    // ...
}
```

### 🔴 问题4：`loadReportItems` 中的关联子查询

**位置**：`internal/handlers/report.go:134-140`

```sql
COALESCE(
    ri.ref_interval_text,
    CASE WHEN ri.ref_interval_id IS NOT NULL THEN
        (SELECT CAST(value_min AS TEXT) || '-' || CAST(value_max AS TEXT)
         FROM reference_intervals WHERE id = ri.ref_interval_id)  -- 隐式 N+1
    ELSE '' END, ''
) as ref_interval_text
```

每行都会触发一次子查询。

**修复方案**：改为 LEFT JOIN：

```sql
SELECT ri.id, ...,
    COALESCE(
        ri.ref_interval_text,
        CASE WHEN ref.id IS NOT NULL
            THEN CAST(ref.value_min AS TEXT) || '-' || CAST(ref.value_max AS TEXT)
        ELSE '' END,
        ''
    ) as ref_interval_text
FROM report_items ri
LEFT JOIN test_items ti ON ti.id = ri.test_item_id
LEFT JOIN reference_intervals ref ON ref.id = ri.ref_interval_id  -- 新增
WHERE ri.report_id = ? ORDER BY ri.id
```

### 🟡 问题5：`DashboardSummary` 的 4 次分散查询

**位置**：`internal/handlers/dashboard.go:14-20`

```go
database.DB.QueryRow(`SELECT COUNT(*) FROM subjects`).Scan(&subjectCount)
database.DB.QueryRow(`SELECT COUNT(*) FROM lab_reports WHERE ocr_status = 'review'`).Scan(&pendingCount)
database.DB.QueryRow(`SELECT COUNT(*) FROM report_items WHERE flag IN ('H', 'L', '阳性', '阴性')`).Scan(&anomalyCount)
database.DB.QueryRow(`SELECT COUNT(DISTINCT hospital_id) FROM lab_reports WHERE hospital_id IS NOT NULL`).Scan(&hospitalCount)
```

**修复方案**：合并为一次查询：

```sql
SELECT
    (SELECT COUNT(*) FROM subjects) AS subject_count,
    (SELECT COUNT(*) FROM lab_reports WHERE ocr_status = 'review') AS pending_count,
    (SELECT COUNT(*) FROM report_items WHERE flag IN ('H','L','阳性','阴性')) AS anomaly_count,
    (SELECT COUNT(DISTINCT hospital_id) FROM lab_reports WHERE hospital_id IS NOT NULL) AS hospital_count
```

## 二、缺失索引

`migrations.go` 中只有 2 个显式索引（均为 `file_md5` 唯一索引）。以下高频查询字段缺少索引：

| 缺失索引 | 影响的查询 | 严重程度 |
|----------|-----------|---------|
| `report_items(report_id)` | `loadReportItems`、`matchRefAndCalcFlag`、`ConfirmReport` | 🔴 极高 |
| `report_items(test_item_id)` | `GetTrendData`、`MatchReference` | 🔴 高 |
| `report_items(flag)` | `DashboardSummary`、`DashboardAnomalies` | 🔴 高 |
| `reference_intervals(test_item_id)` | `MatchReference`（每次 confirm 被调用 N 次） | 🔴 高 |
| `lab_reports(subject_id)` | `ListReports`、`GetTrendData` | 🟠 中 |
| `lab_reports(ocr_status)` | `ListReports`、`DashboardSummary`、`GetTrendData` | 🟠 中 |
| `lab_reports(sample_date)` | `GetTrendData` 的日期范围过滤 | 🟠 中 |
| `test_items(standard_name)` | `MatchTestItemByName` 步骤1、3 | 🟠 中 |
| `test_item_aliases(alias_name)` | `MatchTestItemByName` 步骤2 | 🟡 低 |

**建议在 `migrations.go` 中补充以下 DDL**（追加到 `ddl` 字符串末尾）：

```sql
CREATE INDEX IF NOT EXISTS idx_report_items_report_id    ON report_items(report_id);
CREATE INDEX IF NOT EXISTS idx_report_items_test_item_id ON report_items(test_item_id);
CREATE INDEX IF NOT EXISTS idx_report_items_flag         ON report_items(flag) WHERE flag != '';
CREATE INDEX IF NOT EXISTS idx_ref_intervals_test_item   ON reference_intervals(test_item_id);
CREATE INDEX IF NOT EXISTS idx_lab_reports_subject       ON lab_reports(subject_id);
CREATE INDEX IF NOT EXISTS idx_lab_reports_status        ON lab_reports(ocr_status);
CREATE INDEX IF NOT EXISTS idx_lab_reports_sample_date   ON lab_reports(sample_date);
CREATE INDEX IF NOT EXISTS idx_test_items_standard_name  ON test_items(standard_name);
CREATE INDEX IF NOT EXISTS idx_aliases_alias_name        ON test_item_aliases(alias_name);
```

> `report_items(report_id)` 最关键——它是几乎所有报告详情操作的 WHERE 条件，没有这个索引会全表扫描 `report_items`。

## 三、事务问题

### 🔴 问题1：`matchRefAndCalcFlag` 全程无事务

**位置**：`internal/handlers/report.go:260-368`

该函数包含 2 次 SELECT + N 次 UPDATE/INSERT，但没有任何事务保护。如果中途失败（进程崩溃、超时），数据会处于半更新状态：部分 `report_items` 有 `flag`，其余的没有。

**修复方案**：将整个函数包裹在事务中：

```go
func matchRefAndCalcFlag(reportID string) {
    tx, err := database.DB.Begin()
    if err != nil { return }
    defer func() {
        if p := recover(); p != nil {
            tx.Rollback()
        }
    }()

    // 所有查询和更新改用 tx.QueryRow / tx.Exec
    // ...

    if err := tx.Commit(); err != nil {
        tx.Rollback()
    }
}
```

---

### 🔴 问题2：`ConfirmBatchImport` 文件写入与 DB 操作不原子

**位置**：`internal/handlers/batch_import.go:191-345`

当前流程：

1. 写入 PDF 文件到磁盘 ✅
2. `INSERT INTO lab_reports` — 失败时 `os.Remove(filePath)` 回滚文件 ✅
3. 循环 INSERT `report_items` — **无事务**，部分失败时报告存在但 items 缺失 ❌
4. 调用 `matchRefAndCalcFlag` — **无事务**，失败时报告状态不一致 ❌

**修复方案**：在 INSERT lab_report 之后立即开启事务，包含所有 `report_items` INSERT 和 `matchRefAndCalcFlag` 调用。若事务回滚，同步删除已写入的 PDF 文件。

---

### 🟠 问题3：`ConfirmReport` 两次 Exec 不在事务中

**位置**：`internal/handlers/report.go:246-255`

```go
database.DB.Exec(`UPDATE report_items SET confidence = 100 WHERE report_id = ?`, id)
database.DB.Exec(`UPDATE lab_reports SET ocr_status = 'imported' WHERE id = ?`, id)
// 若第一个成功、第二个失败，状态不一致
```

**修复方案**：两个 Exec 放入同一事务。

## 四、未关闭的 sql.Rows

### 🟠 `matchRefAndCalcFlag` 手动调用 `rows.Close()`

**位置**：`internal/handlers/report.go:307`

```go
rows, err := database.DB.Query(`SELECT ... FROM report_items WHERE report_id = ?`, reportID)
// ...
for rows.Next() { ... }
rows.Close()  // 手动关闭，非 defer
```

若 `rows.Next()` 内部发生 panic，`rows.Close()` 不会执行，导致连接泄漏。

**修复**：改为 `defer rows.Close()`：

```go
rows, err := database.DB.Query(...)
if err != nil { return }
defer rows.Close()
```
