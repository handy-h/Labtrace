报告单管理与批量导入功能

溯源健康 (LabTrace) - 个人纵向检化验管理平台

版本：V1.0 状态：已实现 日期：2026-05-17 关联模块：报告管理、数据导入

1. 功能概述

核心目标：提供独立的报告单查看和管理页面，支持批量导入预先处理好的 JSON + PDF 文件，无需通过 OCR 识别即可直接入库。

本次更新新增两个核心功能模块：

  • UI报告单管理 - 提供独立的报告单查看和管理页面，支持查看所有已入库的检验报告和影像报告，并提供完整的筛选功能
  • 新功能批量导入 - 支持 JSON + PDF 文件批量导入报告数据，支持任意 JSON 格式，用户可自定义字段映射

2. 报告单管理功能

2.1 功能入口

导航栏新增「报告单」菜单项（📄图标）。

2.2 报告类型切换

支持在「检验报告」和「影像报告」之间切换查看。

2.3 筛选功能

检验报告筛选条件

筛选字段 说明
受检者 下拉选择
医院 下拉选择
分类 下拉选择（生化、血常规等）
状态 待处理 / 处理中 / 待核效 / 已入库 / 失败
日期范围 开始日期 ~ 结束日期

影像报告筛选条件

筛选字段 说明
受检者 下拉选择
医院 下拉选择
类型 CT / MRI / X光 / 超声 / 心电图 / 其他
状态 待处理 / 处理中 / 待核效 / 已入库 / 失败
日期范围 开始日期 ~ 结束日期

2.4 报告详情查看

点击列表中的「查看」按钮打开详情弹窗，采用左右分栏布局：

  • 左侧（45%）：原始文件预览
    • 支持图片格式（JPG/PNG）
    • 支持 PDF 格式（内嵌预览）
    • 自动适配文件类型
  • 右侧（55%）：格式化数据展示
    • 基本信息（受检者、日期、医院、分类等）
    • 检验项目表格（项目名、结果、参考区间、单位、提示符）

3. 批量导入功能

3.1 功能入口

导航栏新增「批量导入」菜单项（📦图标）。

3.2 JSON 文件格式说明

重要：系统支持任意 JSON 格式，不要求固定结构。用户通过字段映射配置，将 JSON 中的字段与系统字段进行对应。

示例格式 1：扁平结构

{
  "sample_date": "2024-01-15",
  "items": [
    { "name": "ALT", "value": "25.6", "unit": "U/L", "range": "9-50" },
    { "name": "AST", "value": "23.2", "unit": "U/L", "range": "15-40" }
  ]
}

示例格式 2：嵌套结构

{
  "deptName": "肾脏病中心门诊(余杭)",
  "doctorName": "徐莹",
  "itemName": "肝肾脂糖电解质测定",
  "content": [
    {
      "name": "总蛋白",
      "value": {
        "result": "71.1",
        "max": "85.0",
        "min": "65.0",
        "unit": "g/L",
        "refStatus": "正常"
      }
    }
  ]
}

字段映射配置

用户在导入时可配置以下映射关系（支持点号路径访问嵌套字段）：

系统字段 JSON 路径示例 说明
采样日期 sample_date 报告采样日期
检验项目名称 name 或 content.name 检验项目名称
结果值 value 或 value.result 检测结果值
单位 unit 或 value.unit 检测单位
参考值下限 min 或 value.min 参考区间最小值
参考值上限 max 或 value.max 参考区间最大值

3.3 文件命名规则

  • JSON 和 PDF 文件需使用相同的基础文件名
  • 示例：report_001.json 和 report_001.pdf
  • 系统会自动根据文件名建立映射关系

3.4 操作流程

采用五步骤向导设计：

┌─────────────┐ │ 步骤 1 │ 选择受检者、医院、分类 └──────┬──────┘ ▼ ┌─────────────┐ │ 步骤 2 │ 上传 JSON 和 PDF 文件 └──────┬──────┘ ▼ ┌─────────────┐ │ 步骤 3 │ 配置字段映射（预览实时更新） └──────┬──────┘ ▼ ┌─────────────┐ │ 步骤 4 │ 确认导入（可编辑采样日期） └──────┬──────┘ ▼ ┌─────────────┐ │ 步骤 5 │ 导入结果展示 └─────────────┘

步骤 1：选择受检者和医院

  • 选择目标受检者（必选）
  • 选择医院（可选）
  • 选择检验项目分类（可选）
  • 点击「下一步」继续

步骤 2：上传文件

  • 分别选择 JSON 文件和 PDF 文件（支持多选）
  • 系统自动匹配同名文件
  • 显示匹配结果和错误提示
  • 点击「上传文件」按钮

步骤 3：配置字段映射

  • 左侧显示示例 JSON 数据
  • 右侧配置字段映射路径（支持点号访问嵌套字段）
  • 实时预览前 5 条数据的映射结果
  • 点击「下一步」继续

步骤 4：确认导入

  • 显示所有待导入的报告列表
  • 可编辑每份报告的采样日期
  • 显示每份报告的检验项目数量
  • 点击「确认导入」执行入库

步骤 5：导入结果

  • 显示成功/失败数量
  • 显示错误详情（如有）
  • 提供「继续导入」和「查看报告」按钮

3.5 数据处理逻辑

  1. 文件去重：根据 PDF 文件 MD5 哈希值检测重复，重复文件不会重复入库
  2. 文件保存：PDF 文件保存到 data/uploads/ 目录，文件名格式为 {MD5前12位}_{基础名}.pdf
  3. 数据入库：检验项目以高置信度（100%）直接入库，状态设为「待核效」
  4. 审计日志：记录每份报告的批量导入操作

4. API 接口变更

新增POST /api/v1/batch/upload

上传批量文件对，返回解析结果。

请求：multipart/form-data

  • json_files：JSON 文件列表
  • pdf_files：PDF 文件列表

响应：

{
  "code": 0,
  "message": "ok",
  "data": {
    "results": [
      {
        "file_name": "report_001",
        "data": { ... },
        "items": [ ... ]
      }
    ],
    "errors": []
  }
}

新增POST /api/v1/batch/confirm

确认并执行批量导入。

请求：

{
  "subject_id": 1,
  "hospital_id": 1,
  "category_id": 1,
  "mappings": {
    "sample_date": "sample_date",
    "item_name": "name",
    "item_value": "value.result",
    "item_unit": "value.unit",
    "ref_min": "value.min",
    "ref_max": "value.max"
  },
  "reports": [
    {
      "file_name": "report_001",
      "data": { ... },
      "pdf_data": "data:application/pdf;base64,..."
    }
  ]
}

响应：

{
  "code": 0,
  "message": "ok",
  "data": {
    "success_count": 5,
    "fail_count": 1,
    "errors": ["report_003: 文件已存在"],
    "report_ids": [10, 11, 12, 13, 14]
  }
}

增强GET /api/v1/reports

新增筛选参数：

  • category_id：分类 ID
  • start_date：开始日期
  • end_date：结束日期

增强GET /api/v1/imaging-reports

新增筛选参数：

  • ocr_status：OCR 状态
  • start_date：开始日期
  • end_date：结束日期

7. 安全性考虑

  • 文件类型验证：仅接受 .json 和 .pdf 文件
  • 文件大小限制：建议在 Web 服务器层面配置
  • 重复检测：通过 MD5 哈希防止重复导入
  • SQL 注入防护：使用参数化查询
