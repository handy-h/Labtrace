🔴 严重问题

6 项
🔴
Goroutine 中数据库操作缺乏错误处理
internal/handlers/ocr.go:90-145
OCR 异步处理 goroutine 中多次调用 database.DB.Exec()，所有返回的 error 均被丢弃。若数据库写入失败（如磁盘满、连接断开），报告状态会停留在 'processing'，用户无法感知错误，且可能造成数据不一致。
修复建议：为 goroutine 内的每个 DB 操作检查错误并记录日志；设置超时 context 防止 goroutine 永久阻塞；考虑使用事务包裹原子操作。
预估修复时间：30分钟
🔴
备份恢复使用固定临时文件路径
internal/handlers/backup.go:49
ImportBackup 使用硬编码路径 /tmp/labtrace_restore.bak。并发请求会导致文件竞争；同一时间两个用户导入备份会造成数据损坏。此外 tmpFile 的创建错误被忽略。
修复建议：使用 os.CreateTemp("", "labtrace_restore_*.bak") 生成唯一临时文件；检查 os.Create 返回的 error。
预估修复时间：10分钟
🔴
os.Exit(0) 跳过 defer，数据库关闭顺序不安全
main.go:148-155
优雅关闭 goroutine 中调用 database.Close() 后立即 os.Exit(0)。由于 os.Exit 不执行任何 defer，且 r.Run() 仍在主 goroutine 运行，存在竞态：可能在连接关闭前就退出，或连接关闭期间 Gin 仍在处理请求。
修复建议：使用 http.Server.Shutdown(ctx) 替代 os.Exit(0)，通过 r.Run 包装为 http.Server，在信号处理中优雅关闭 HTTP 服务后再关闭数据库。
预估修复时间：30分钟
🔴
CORS 中间件允许所有来源
internal/middleware/cors.go:9
Access-Control-Allow-Origin: * 允许任意域名的跨域请求。虽然本项目是单体应用，但 API 无认证保护的情况下，任何网站都可以调用 API 读取/删除用户数据。
修复建议：将 CORS 配置为仅允许同源请求（不设置 Allow-Origin 头），或从环境变量读取允许的 Origin 白名单。同时添加 CSRF 保护机制。
预估修复时间：15分钟
🔴
趋势查询存在 N+1 查询问题
internal/services/trend_service.go:101-106
GetTrendData 在遍历每个 report_item 时，如果 refID.Valid，会对每个点执行一次 SELECT value_min, value_max FROM reference_intervals 查询。当报告积累到数百条时，单次趋势查询可能触发数百次额外 SQL 查询，严重影响性能。
修复建议：将参考区间查询改为 JOIN 或在主查询中使用子查询一次性获取；或批量收集所有 refID 后用 WHERE id IN (...) 一次查询。
预估修复时间：20分钟
🔴
API 无任何认证/授权机制
main.go:50-145
所有 API 端点完全开放，无认证、无鉴权。任何能访问 8080 端口的人都可以查看所有受检者数据、上传/删除报告、导出/导入备份、修改 OCR 配额。备份文件虽然加密，但数据库本身未加密。
修复建议：短期：至少添加基于 token 的简单认证中间件；长期：集成 JWT 或 session 认证，支持多用户隔离。同时考虑对敏感操作（备份导入、删除）添加确认机制。
预估修复时间：2小时

缺少全局错误恢复中间件
internal/middleware/
项目没有 panic recovery 中间件。Gin 默认的 gin.Default() 包含 Logger 和 Recovery，但如果在自定义中间件或 goroutine 中发生 panic，可能导致整个进程崩溃。
优化建议：确认 Gin 默认的 Recovery 中间件已生效；为 goroutine 添加 defer recover() 防止未处理的 panic。
预估修复时间：10分钟

ECharts 实例未使用响应式更新
web/js/components/sparkline.js:53-56
Sparkline 组件在数据变化时先 dispose() 再重新 initChart()，而非使用 setOption 增量更新。对于频繁更新的场景会造成不必要的 DOM 操作和闪烁。
优化建议：改为使用 chartInstance.setOption(newOption, true) 进行增量更新，仅在容器尺寸变化时才重新初始化。
预估修复时间：10分钟
